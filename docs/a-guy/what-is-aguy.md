## **1\. מה זו מערכת aguy באמת (בלי רומנטיקה)**

aguy היא **פלטפורמת למידה ותודעה מונעת-AI**, שנבנית כ־**מערכת הפעלה תוכניתית** (Operating System), לא כאתר ולא כ־LMS קלאסי.

זו מערכת אחת שמאחדת:

* מערכת לימוד מתמטית עמוקה (קורסים, פרקים, שיעורים, תרגילים)
* CMS כללי לניהול תוכן, עמודים, פרומפטים וקונפיגורציה
* מערכת צ’אט חכמה עם הקשר, זיכרון ו-Tutor
* תשתית סוכנים (Code / Docs / Automations)
* Multi-Tenant \+ Multi-Locale כתשתית, לא כתוספת

זו **לא מערכת תוכן**.
זו **מערכת שמנהלת ידע, הקשר ופעולה**.

---

## **2\. הסטאק (ומה כבר ננעל)**

### **Core Stack**

* **Payload CMS** – הליבה. לא “CMS”, אלא מנוע:
  * Data Models
  * Admin UX
  * Access Control
  * Config-as-Data
* **Next.js (App Router)** – פרונט ואדמין, מופרדים בבירור
* **MongoDB Atlas** – כולל Vector Search (אמיתי, לא צעצוע)
* **Tailwind \+ shadcn/ui** – שפה עיצובית אחידה, ללא CSS ספגטי
* **Vercel** – דומיינים, סביבות, פריסה

❗ תיקון מהותי:
אין Radix כ-Dependency ישיר.
המערכת שומרת מינימליזם טכנולוגי מכוון.

הבחירה כאן עקבית:
**מעט כלים, שליטה מלאה, עומק אמיתי.**

## **3\. עקרון־על ארכיטקטוני (מעודכן)**

### **“Everything is Data — incrementally.”**

החזון הוא **CMS-Driven**, אבל הביצוע **אינקרמנטלי, זהיר, ומדורג**.

בפועל:

* יש עדיין עמודים וזרימות קשיחות
* יש בלוקים שלא ממומשים
* יש קוד UI שעדיין מחוץ ל-CMS

וזה **בסדר**.

העיקרון הנכון:

* לא שוברים מערכת עובדת
* לא עושים Rewrite אידאולוגי
* כל שלב מקרב לעוד יכולת CMS-Driven

זו בגרות הנדסית, לא חולשה.

## **4\. מערכת התוכן והלמידה (LMS)**

### **היררכיית ליבה**

* Course
* Chapter
* Lesson
* Exercise

ה-Exercise הוא **האטום הבסיסי** של המערכת.

תרגיל יכול להיות:

* מוזן מ-PDF
* מבוסס LaTeX
* מפורק ל-Context Blocks
* מנוהל כישות AI-Aware

ה-PDF הוא **לא יעד**.
הוא **מקור פירוק זמני**.

## **5\. PDF & Math Pipeline (יתרון תחרותי אמיתי)**

* PDF.js משולב Inline
* Highlight / Annotation
* Extraction באמצעות מודל Vision
* Chunking ל-MemoryItems
* אחסון לפי רמות הקשר:
  * Lesson Context
  * Exercise Context
  * Diagram Context (עתידי)

רוב המערכות עוצרות ב-“העלאת PDF”.
אתה משתמש ב-PDF כ־Raw Material ל-AI.

זו הבחנה קריטית.

## **6\. AI & Memory Architecture**

### **שכבות זיכרון**

* Session Memory
* Task / Interaction History
* MemoryItems (עם Embeddings)
* Vector Search ב-MongoDB

### **שימושים בפועל**

* Tutor בצ’אט
* פתרון תרגילים
* חיבור בין תוכן ללומד
* בסיס לאוטומציות עתידיות

אתה **בונה מערכת זיכרון ייעודית**,
אבל עם ריסון: בוחן ספריות קיימות לפני פיתוח עצמי.

זה נכון. אחרת זה בור בלי תחתית.

## **7\. מערכת הצ’אט (לא “בוט”)**

* Gemini / OpenAI לפי צורך
* Payload שולט בקונטקסט
* העלאת מדיה (Images \+ PDF)
* Retention מדוד (ephemeral)
* AIUsage tracking: טוקנים, עלות, latency

הצ’אט הוא **ממשק על**,
לא פיצ’ר צד.

הוא מחבר בין:
תוכן ← זיכרון ← משתמש ← פעולה.

## **8\. Automations & Rules (אזור סיכון מודע)**

החזון:

* Trigger
* Condition
* Action
* State

המציאות:

* אין רצון לבנות n8n פנימי
* אין ROI מיידי למנוע כבד

הכיוון הנכון:

* מינימום לוגיקה פנימית
* מקסימום האצלה לכלים חיצוניים
* אינטגרציה, לא חיקוי

## **9\. Multi-Tenant & Locale**

* Tenant הוא First-Class
* ConfigEntries scoped ל-Tenant
* אין Global Fallback (בכוונה)
* Locale בעיקר על Course, לא על כל העולם

זה מצמצם מורכבות **איפה שצריך**
ולא מפזר אותה אידאולוגית.
